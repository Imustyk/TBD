# Проектный план

## Тема проекта

**«Читай и заставь их читать»**  
Информационная система для сбора, хранения, анализа, визуализации и прогнозирования читательской активности.

## Назначение документа

Этот файл является расширенным описанием проекта. Он резюмирует:

- текущее состояние репозитория
- уже реализованные компоненты
- основные требования к итоговой системе
- целевую архитектуру
- функциональные возможности приложения
- план развития проекта от учебного прототипа к более реалистичной версии

Документ можно использовать как основу для:

- пояснительной записки
- технического задания
- плана дипломного проекта
- roadmap дальнейшей разработки

## Идея проекта

Проект `«Читай и заставь их читать»` предназначен для анализа поведения читателей и прогнозирования интереса к книгам. Система должна помогать отслеживать:

- какие книги становятся популярными
- как меняется активность пользователей
- какие жанры читают чаще
- как на интерес к книге влияют рейтинг, отзывы и история чтения
- какие книги с высокой вероятностью будут прочитаны пользователем

Итоговая версия проекта должна быть ориентирована не только на демонстрацию ETL и машинного обучения, но и на работу с реальными данными, получаемыми через `API` и/или веб-скраппинг.

## Анализ текущей структуры проекта

На данный момент в каталоге `project/` уже есть рабочий учебный прототип.

### Основные папки и их назначение

- `app/`
  Локальное веб-приложение на `Streamlit`, отображающее аналитику и прогнозы.
- `data/`
  Хранение исходных, промежуточных и итоговых данных.
- `etl/`
  Конвейер извлечения, преобразования и загрузки данных.
- `dwh/`
  SQL-схема хранилища данных.
- `ml/`
  Обучение моделей и запуск предсказаний.
- `models/`
  Сохранённые модели и метрики обучения.

### Основные файлы

- `app/streamlit_app.py`
  Веб-интерфейс с аналитикой и предсказаниями.
- `etl/extract.py`
  Получение данных из `SQLite` и `MongoDB`.
- `etl/transform.py`
  Очистка данных, агрегации, расчёт признаков, подготовка витрин.
- `etl/load.py`
  Загрузка данных в локальное DWH.
- `dwh/schema.sql`
  Структура таблиц `fact` и `dimension`.
- `ml/train_xgboost.py`
  Модель прогнозирования популярности книги.
- `ml/train_lightgbm.py`
  Модель прогнозирования читательской активности.
- `ml/predict.py`
  Модуль предсказания на основе сохранённых моделей.
- `data/generate_sample_data.py`
  Генератор демонстрационных данных.
- `README.md`
  Краткое описание проекта и инструкции по запуску.

## Ограничения текущей версии

Несмотря на рабочее состояние, текущая версия пока остаётся учебной и демонстрационной.

### Главные ограничения

- используются синтетические, а не реальные данные
- интерфейс частично не русифицирован
- веб-интерфейс показывает аналитику, но почти не управляет процессами
- нет полноценного получения данных из внешних источников в реальном времени
- нет отдельного слоя скраппинга сайтов
- нет накопления исторических снапшотов данных
- нет стратегии автоматического дообучения моделей
- нет учёта версий датасетов и версий моделей
- нет системы логирования этапов загрузки, обучения и ошибок на уровне UI

## Целевая концепция проекта

Итоговая версия системы должна быть не просто аналитической витриной, а полноценной локальной информационной системой с интерфейсом управления.

### Ключевая цель

Создать русскоязычное веб-приложение, которое:

- собирает данные о книгах и читательской активности через `API` и/или скраппинг
- хранит исходные данные, промежуточные данные и исторические версии
- обновляет хранилище данных по расписанию или по команде пользователя
- обучает и переобучает модели на новых данных
- отображает аналитику, графики и прогнозы через локальный веб-интерфейс
- позволяет пользователю запускать процессы через интерфейс сайта

## Основные требования к проекту

### Функциональные требования

- приложение должно иметь интерфейс на русском языке
- приложение должно запускаться локально как веб-сайт
- приложение должно визуализировать метрики, тренды и прогнозы
- приложение должно получать данные из внешних источников
- приложение должно поддерживать `API ingestion`
- приложение должно поддерживать веб-скраппинг
- приложение должно сохранять сырые данные отдельно от очищенных
- приложение должно накапливать новые данные, а не только перезаписывать старые
- приложение должно позволять запускать ETL и обучение модели из интерфейса
- приложение должно отображать статус загрузки данных, ETL и обучения
- приложение должно строить прогноз популярности книг
- приложение должно строить прогноз вероятности чтения книги пользователем
- приложение должно сохранять результаты моделей и ключевые метрики качества

### Нефункциональные требования

- система должна быть понятной для демонстрации на защите
- структура проекта должна быть модульной и расширяемой
- данные должны храниться в компактном и удобном формате
- должна быть возможность быстро заменить демонстрационные данные реальными
- система должна работать локально без обязательного облачного развёртывания
- код должен быть разделён на независимые слои: ingestion, ETL, DWH, ML, UI

## Целевая архитектура

Итоговая версия проекта должна состоять из следующих уровней:

### 1. Источники данных

- книжные API
- сайты с книгами и отзывами
- каталоги библиотек
- площадки с рейтингами
- новостные и тематические ресурсы о книгах

### 2. Слой сбора данных

На этом уровне реализуются:

- запросы к внешним `API`
- парсинг HTML-страниц
- извлечение метаданных о книгах
- извлечение рейтингов, отзывов, количества оценок
- регулярное обновление данных

Для этого слоя можно использовать:

- `requests`
- `httpx`
- `BeautifulSoup`
- `lxml`
- `playwright` или `selenium`, если сайт динамический

### 3. Слой хранения операционных данных

Предлагаемое разделение:

- `SQLite` или `PostgreSQL`
  для структурированных данных:
  пользователи, книги, история чтения, рейтинги, события
- `MongoDB`
  для неструктурированных данных:
  отзывы, комментарии, рекомендации, текстовые описания, результаты парсинга
- файловое хранилище
  для сырых JSON-снимков и логов загрузки

### 4. ETL / ELT слой

Должен выполнять:

- извлечение новых данных
- дедупликацию
- очистку ошибок и пропусков
- унификацию форматов дат, категорий и названий
- расчёт агрегатов
- подготовку признаков для моделей
- загрузку в хранилище данных

### 5. Data Warehouse

Хранилище должно содержать:

- `fact_reading_activity`
- `dim_user`
- `dim_book`
- `dim_time`
- `dim_category`

При расширении проекта можно добавить:

- `dim_source`
- `dim_author`
- `fact_review_activity`
- `fact_book_snapshot`

### 6. Слой машинного обучения

Должны поддерживаться:

- первичное обучение моделей
- переобучение на новых данных
- сравнение качества новых и старых моделей
- хранение артефактов модели
- хранение метрик качества

Основные модели:

- `XGBoost Regressor` для прогноза популярности книги
- `LightGBM Regressor` для прогноза читательской активности

### 7. Веб-интерфейс

Интерфейс должен быть русскоязычным и содержать не только аналитику, но и элементы управления системой.

Проект должен хранить данные в нескольких формах.

### 1. Raw layer

Сырые данные в исходном виде:

- ответы `API`

### 2. Staging layer

Очищенные и нормализованные данные:

- унифицированные таблицы книг

- отзывы
- история чтения

Агрегированные таблицы и витрины для аналитики и ML.

### 4. Model artifacts

Должны сохраняться:

- обученные модели
- метрики качества

- версия датасета
- версия модели

## Подход к работе с реальными данными

Для защиты проекта особенно полезно показать, что система умеет не только работать на синтетике, но и получать данные извне.

### Вариант 1. API-first

Система получает данные через официальные `API`, если они доступны.

Плюсы:

- данные структурированы
- меньше проблем с изменением HTML
- проще поддерживать стабильность

### Вариант 2. Scraping-first

Если готового `API` нет, система получает данные со страниц сайтов.

Плюсы:

- можно брать данные почти с любого открытого источника
- гибкость по выбору площадок

Минусы:

- выше вероятность поломки при изменении верстки
- нужно учитывать legal и ethical ограничения источника

### Рекомендуемый подход

Лучше сделать гибридную схему:

- при наличии `API` использовать его как основной источник
- при отсутствии `API` подключать модуль скраппинга
- все полученные данные сохранять в `raw`-слой с отметкой времени и источника

## Хранение истории и накопление данных

Одно из главных требований будущей версии проекта: данные не должны просто перезаписываться.

Система должна:

- хранить каждую новую выгрузку с временной меткой
- сохранять историю рейтингов и отзывов
- сохранять историю изменений популярности книги
- уметь сравнивать новое состояние со старым
- поддерживать накопление обучающей выборки

### Что желательно добавить в проект

- папку `data/raw/` для сырых выгрузок
- таблицу журналов загрузки
- таблицу источников данных
- таблицу снапшотов книг и рейтингов
- хранение версии датасета для каждого обучения

## Переобучение модели

Итоговая система должна поддерживать постоянное улучшение анализа.

### Минимально необходимая логика

- загрузить новые данные
- обновить staging и DWH
- сформировать новый набор признаков
- обучить новую версию модели
- сравнить метрики со старой версией
- сохранить лучшую модель

### Варианты запуска

- вручную из интерфейса
- по расписанию
- автоматически при накоплении достаточного числа новых записей

## Риски проекта

- некоторые сайты могут не иметь открытого `API`
- структура HTML у сайтов может меняться
- реальных данных может оказаться мало для качественного обучения
- переобучение без контроля качества может ухудшить модель
- синтетические данные могут создавать слишком красивую картину в демо

## Критерии готовности проекта

Проект можно считать хорошо подготовленным для защиты, если:

- интерфейс полностью русскоязычный
- локальный веб-интерфейс запускается и работает стабильно
- данные загружаются из внешнего `API` или через скраппинг
- хранятся сырые, очищенные и агрегированные данные
- ведётся история обновлений
- модели обучаются и переобучаются на новых данных
- визуализация показывает не только статические графики, но и текущее состояние системы
- пользователь может управлять ключевыми этапами через интерфейс

## Итоговое резюме

На данный момент проект уже представляет собой хороший учебный прототип с `ETL`, `DWH`, `ML` и `Streamlit`. Однако для того, чтобы он соответствовал твоей теме и выглядел сильнее на защите, его нужно развить в сторону:

- русскоязычного интерфейса
- реальных источников данных
- хранения исторических снапшотов
- управления процессами через веб-интерфейс
- переобучения моделей на новых данных

Именно в таком виде проект будет восприниматься не как просто демонстрация алгоритмов, а как полноценная информационная система.
